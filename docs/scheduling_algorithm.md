# 计划生成与动态调整算法

## 输入

计划算法只依赖纯数据输入：

- 用户目标 `UserGoal`。
- 去重后的 `WordItem` 集合。
- 历史 `DailyTask`。
- 历史 `ProgressRecord`。
- 当前重排起点日期 `asOfDate`。
- 新计划版本号。

界面组件不直接包含调度逻辑，核心函数可被 Vitest 独立调用。

## 日期与学习日

1. 枚举 `asOfDate` 到 `deadline` 的所有自然日。
2. 根据 `restWeekdays` 标记休息日。
3. 非休息日为剩余有效学习日。
4. 按 `bufferDayRatio` 从学习日末尾预留缓冲日。
5. 初始计划优先把新学任务分配到非缓冲学习日；当任务量较大时允许使用缓冲日，但必须在原因中说明。

## 实际待学习量

实际待学习量由用户总目标减去历史新学完成量得到：

```text
remainingNewWords = max(targetVocabularyCount - sum(progress.newWordsCompleted), 0)
```

这保证少学不会减少总目标，超额完成会真实降低剩余量。历史记录不会被重排改写。

## 可行性判断

```text
requiredDailyAverage = ceil(remainingNewWords / remainingEffectiveDays)
```

当 `requiredDailyAverage > dailyNewWordLimit` 时，状态为 `infeasible`，必须显示：

- 剩余待学习词数。
- 剩余有效学习日。
- 当前每日新学上限。
- 按期完成所需最低每日新学量。
- 超出上限的数量。
- 不可行原因。

当剩余量为 0，状态为 `completed`。当任务需要使用缓冲日或接近日上限时，状态为 `atRisk`。其余为 `feasible`。

## 新学任务分配

### 平稳型

尽量平均分配到可用学习日，日任务差值控制在 1 以内，且不超过每日上限。

### 前紧后松型

使用前高后低的权重分配任务，但任意一天不得超过每日上限。

### 弹性型

初始接近平稳分布；动态重排时允许未来任务略有波动，并记录调整原因。

## 复习任务分配

第一版采用固定间隔复习节点：

- 新学后第 1 天。
- 第 3 天。
- 第 7 天。
- 第 14 天。
- 第 30 天。

系统根据历史实际新学量和未来计划新学量生成复习数量，并尽量把超出 `dailyReviewLimit` 的复习任务顺延到后续学习日。复习和新学分别统计。

## 动态调整

每日记录后，系统从次日开始重排未来任务：

- 少完成：未完成的新词仍留在总剩余量中，被分配到后续学习日，优先使用缓冲日。
- 超额完成：超出的新词计入实际进度，后续剩余任务减少。
- 未学习：当天新学任务仍未完成，复习任务视为延期；系统重新判断目标是否可行。

动态调整只改变未来每日新学数量、未来复习安排、周/月任务分布和缓冲日使用，不修改总目标、截止日期、词书范围或历史完成记录。

## 月度与周度汇总

计划生成后按日期聚合：

- 月度：本月计划新学、计划复习、缓冲天数、实际完成、累计完成比例。
- 周度：本周计划新学、计划复习、每日分布、完成率、欠缺任务。
- 长期：总目标、已完成、剩余、剩余有效学习日、可行状态和阶段。

## 边界处理

- 截止日期早于开始日期：目标校验失败。
- 没有剩余有效学习日且仍有剩余任务：不可行。
- 每日上限小于等于 0：目标校验失败。
- 连续多日未完成：剩余任务增加压力，但总目标不减少。
- 休息日不安排新学任务，复习溢出也尽量跳过休息日。
