# AGENTS.md

## 项目名称

智能背单词计划安排器（Vocabulary Smart Planner）

## 总体工作方式

这是一个全新的独立软件开发项目。请只基于当前项目目录中的文件开展工作，不读取、不引用、不延续其他项目中的旧代码、旧日志或旧设计。

请首先完整读取本文件和 `TASK_FULL.md`，理解产品目标、范围、算法约束和验收标准。

不要只输出开发建议或空泛方案。请在完成必要设计文档后，继续实施可运行的 MVP 代码，并持续运行测试验证核心逻辑。

## 核心原则

本项目的重点不是制作一个功能庞杂的背单词应用，而是制作一个可靠的：

1. 总目标管理系统；
2. 词书推荐与导入系统；
3. 长期、月度、周度、每日计划生成系统；
4. 根据完成情况动态重排任务的调度系统；
5. 可离线使用、可备份、可解释的学习计划工具。

任何功能扩展都不得削弱以上核心能力。

## 第一版技术栈

默认采用以下技术方案：

- React
- TypeScript
- Vite
- IndexedDB，优先使用 Dexie 管理本地数据
- Vitest 进行单元测试
- CSS 或轻量级组件方案完成基础界面

第一版为本地离线 Web 应用，不实现：

- 用户登录；
- 云同步；
- 多端同步；
- 外部大模型 API；
- 自动抓取网络词库；
- 发音播放；
- 例句生成；
- 单词拼写训练；
- 社交、排行榜、打卡分享功能。

## 数据与版权约束

- 不要未经许可抓取商业词书完整内容。
- 软件可以内置演示词表和词书元数据。
- 正式词表内容采用 CSV 或 JSON 导入方式。
- 必须提供清晰的导入模板和演示数据。
- 同一单词来自多个词书时，必须支持去重和来源标记。

## 工程质量要求

- 所有核心业务逻辑使用 TypeScript 类型约束。
- 计划算法与界面组件分离，不要把调度逻辑直接写在页面组件内。
- 核心算法必须可以被单元测试独立调用。
- 每个关键函数需要说明输入、输出和约束。
- 对用户输入进行校验。
- 所有日期处理必须明确时区和日期边界，不得因跨天造成任务错乱。
- 目标任务不得因动态调整而被静默删除。
- 用户数据不得在刷新页面后丢失。
- 导入失败时必须给出明确错误原因。
- 软件界面以中文为默认语言。

## 动态调整的最高约束

动态调整只能改变：

- 后续每日新学词数量；
- 后续复习任务安排；
- 周计划和月计划内的任务分布；
- 缓冲日的使用方式。

动态调整不得静默改变：

- 用户设定的最终截止日期；
- 用户设定的总目标词汇范围；
- 用户选择的词书范围；
- 已完成学习记录。

当剩余任务已经无法在用户限定的每日最大任务量和截止日期内完成时，系统不得伪造可完成状态，必须显示：

- 当前剩余任务量；
- 剩余有效学习天数；
- 按原目标完成所需的最低每日新学词量；
- 与用户当前每日上限的差距；
- 当前计划状态为“按现有限制无法完成”。

## 测试要求

至少覆盖以下测试情境：

1. 正常生成长期、月度、周度和每日计划。
2. 用户按计划完成任务后，剩余任务正确减少。
3. 用户某一天未完成任务，欠缺任务被合理分配到后续日期。
4. 用户超额完成任务，后续任务量合理降低。
5. 用户连续多天未完成任务，不允许总目标被静默缩减。
6. 剩余学习天数不足时，系统正确判断计划不可行。
7. 每日任务上限生效。
8. 休息日设置生效。
9. 多词书导入后单词去重正确。
10. 导出后重新导入，用户目标和完成记录能够恢复。

## 完成标准

本轮工作的完成标准不是“已经创建项目文件”，而是：

1. 软件可以本地启动；
2. 用户可以输入目标并生成计划；
3. 用户可以查看今日、每周、每月和长期任务；
4. 用户可以记录每天完成情况；
5. 软件可以基于记录重新安排后续任务；
6. 重新安排不改变总目标；
7. 关键算法通过自动化测试；
8. 有清晰的使用说明和工作日志。
