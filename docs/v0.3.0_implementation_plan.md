# v0.3.0 实施计划

## 阶段一：远程核查与基线

- 拉取远程分支和 tag。
- 核对默认分支、v0.1.0 tag、v0.2.0、v0.2.1、v0.3.0 状态。
- 切换 v0.2.1 分支，运行测试和构建。
- 创建并推送 `snapshot/v0.2.1-technical-baseline-before-v0.3.0`。
- 从 v0.2.1 创建 `codex/v0.3.0-smart-vocab-beta`。

## 阶段二：设计冻结

- 创建 v0.3.0 产品、流程、模型、调度、AI、迁移、验收和实施计划文档。
- 更新 README、CHANGELOG、WORKLOG、AGENTS、TASK_FULL。
- 提交 `docs: define v0.3.0 product architecture and acceptance contract`。

## 阶段三：核心结构和迁移

- 扩展 `src/domain/types.ts`。
- 升级 Dexie v3 表结构和迁移。
- 扩展备份 schema v3。
- 增加迁移与备份测试。

## 阶段四：目标规划与词书匹配

- 增加目标版本历史和阶段计划生成。
- 重构自然语言建议为 AI 服务接口加本地规则 provider。
- 扩展词书状态、启用机制和覆盖统计。
- 改进具体学习范围选择优先级。

## 阶段五：学习执行与历史

- 整理今日任务工作台。
- 完善新词和复习流程状态展示。
- 增加词条详情和学习历史。
- 增加重点遗忘词识别。
- 保持 v0.2.1 逐词和结算回归测试。

## 阶段六：动态计划与复盘

- 汇总长期、阶段、月、周、日计划。
- 实现周度复盘和月度复盘数据。
- 扩展风险和不可行说明。
- 重排时写入可读调整日志。

## 阶段七：AI 安全降级

- 实现本地规则建议、AI 接口、结构校验、失败降级。
- 建议展示、影响预览、用户确认和应用记录。
- 不提交 API Key，不在备份和日志写入密钥。

## 阶段八：验证与发布准备

- 更新所有文档和 Release Notes。
- 运行 `npm.cmd test`。
- 运行 `npm.cmd run build`。
- 若存在 `lint` 或 `typecheck` 脚本则运行。
- 推送 v0.3.0 分支。

## 阶段九：发布尝试

具备权限时：

- 合并到默认主分支。
- 创建并推送 `v0.3.0` tag。
- 创建 GitHub Release。

若当前环境无 Release 工具或权限：

- 不声称正式发布完成。
- 保留 `docs/v0.3.0_release_notes.md`。
- 报告剩余 GitHub 操作。
