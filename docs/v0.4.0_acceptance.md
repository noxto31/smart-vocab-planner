# v0.4.0 验收标准

## 自动化测试要求

必须通过：

```bash
npm.cmd test
npm.cmd run build
```

同时继续通过 v0.3.0 核心回归：

- 具体单词学习任务。
- 具体复习任务。
- 当日结算。
- 跨日结算。
- 词库不足与用户未完成区分。
- 备份导出与恢复。
- 旧版本迁移。
- AI 建议不自动应用。

## 动态恢复

| 场景 | 输入 | 预期 |
| --- | --- | --- |
| 漏学一天后的平滑分摊 | 目标 1000，新词 10 天，第 1 天完成 0 个 | 产生 100 个待补学任务；剩余 9 天每日约 111 至 112 个；第 2 天不得变成 200 个；目标和截止日期不变 |
| 少完成一部分后的平滑分摊 | 第 1 天计划 100 个，完成 60 个 | 产生 40 个待补学任务；后续日期小幅增加；不得集中堆到某一天 |
| 复习高峰日避让 | 100 个待补学新词，未来某天有大量复习 | 补学优先分配到总负荷较低日期，不机械压到复习高峰日 |
| 每日上限不足 | 剩余新词 1000，剩余学习日 9，每日新词上限 105 | 显示无法按期完成，最低日均约 112；不得减少目标或延长截止日期 |
| 词库不足但完成可执行词 | 目标大于已导入词表，用户完成全部可执行词 | 词表缺口保留，不计为学习失败 |

## 计划视图

| 视图 | 验收点 |
| --- | --- |
| 今日任务卡片 | 显示原计划、调整后计划、调整原因、今日新词、复习、待补学、逾期复习和任务压力 |
| 周计划卡片 | 显示 7 天任务分布、压力状态、休息日、动态调整标记和增加原因 |
| 月历 / 热力图 | 按日期聚合新词、复习、总负荷、压力状态、完成/未完成、休息和补学标记 |
| 阶段时间轴 | 显示阶段范围、目标、已完成、剩余、状态和阶段命名说明 |
| 详细明细 | 仍可查看每日表格，但不是唯一计划展示 |

## 中文用词

| 场景 | 预期 |
| --- | --- |
| 内部状态显示 | 页面中不得直接显示 `ok`、`near_limit`、`over_limit`、`planned`、`reviewing`、`mastered`、`inventoryGap`、`backlog` 等内部状态 |
| 核心指标 | 使用“本期目标词数”“已导入词汇总数”“待补学”“任务压力”等统一中文表达 |
| 状态映射 | 状态翻译来自统一工具函数或映射文件 |
| 调整原因 | 使用自然中文解释，不展示内部枚举 |

## 六级目标建议

| 场景 | 预期 |
| --- | --- |
| 用户输入六级自然语言目标但未导入真实词表 | 不得直接生成 900 词作为正式建议 |
| 没有真实词表或基础测评 | 提示先导入目标词表或使用演示模式 |
| 显示参考范围 | 必须标明只是参考范围，不是最终计划词量 |
| 应用建议 | 用户确认前不得改变正式目标或生成可执行计划 |

## 发布验收

开发完成后：

1. 推送 `codex/v0.4.0-dynamic-recovery-calendar-ui`。
2. 检查可接受后合并到 `master`。
3. 创建 `v0.4.0` tag。
4. 创建 GitHub Release：`v0.4.0 - 动态进度恢复与计划视图重构版`。

如果当前环境无法创建 Release：

- 至少推送分支。
- 如已完成合并和 tag，说明 Release 仍需手动创建。
- 生成 `docs/v0.4.0_release_notes.md`。
- 不声称已经正式发布。
